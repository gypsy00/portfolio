The Problem

I wanted to understand how real-time communication actually works under the hood. Not just sending messages, but everything users expect from a modern chat app — knowing when someone's typing, seeing when your message has been read, and knowing who's online right now.

Rather than using a pre-built chat service, I built every layer myself to understand the full architecture: authentication, real-time event handling, presence management, and persistent message storage.

How It Works

The app runs on a single Node.js server handling both HTTP requests and WebSocket connections on the same port. Users authenticate via REST endpoints and receive a JWT token, which is then used to authenticate the WebSocket connection. All real-time features flow through Socket.io event handlers organised into separate concerns.

Client (Vanilla JS) ↓ REST API for auth + room management ↓ WebSocket for real-time events Express Server (single port) ├── chatHandler → messages, read receipts ├── roomHandler → create, join, switch rooms └── presenceHandler → online/offline, typing MongoDB ├── Users (auth credentials) ├── Rooms (members, metadata) └── Messages (content, read receipts)

Key Features

Real-Time Messaging Messages appear instantly for all room members via WebSocket events. No polling, no delays.
Read Receipts Per-message tracking of who has read what. When a user views a message, all senders in the room are notified in real-time.
Typing Indicators Live "user is typing..." indicators that fire over WebSockets — no database writes, purely event-driven.
Presence System Tracks who's online and offline. Multi-tab aware — a user isn't marked offline until all their tabs are closed.
Multi-Room Support Users can create rooms, join existing ones, and switch between them. Each room maintains its own message history and member list.
Unread Badges & Pagination Unread message counts with visual badges per room. Message history loads with cursor-based pagination on scroll.
Auto-Reconnection If the connection drops, the client automatically reconnects and shows a status indicator so users know what's happening.

Tech Stack

Node.js + Express 5 Server runtime and REST framework. Single process handles both HTTP and WebSocket on port 3000.
Socket.io 4 WebSocket abstraction with automatic fallback, room management, and namespace support.
MongoDB + Mongoose 9 Document database for users, rooms, and messages. Schema validation with Mongoose models.
JWT + bcrypt Token-based auth for both REST and WebSocket layers. Passwords hashed with bcrypt before storage.
Vanilla JavaScript Zero-dependency frontend served statically. Proves understanding of DOM manipulation without framework abstractions.
HTML / CSS Custom responsive layout with status indicators, badges, and room switching UI.

What I Learned

Building this from scratch taught me how bi-directional communication really works — the difference between request-response (REST) and event-driven (WebSocket) patterns, and when to use each. REST handles auth and room CRUD; WebSockets handle everything that needs to be instant.

The presence system was the most interesting challenge. A user might have multiple tabs open, so I couldn't just mark them offline when one tab closes. I had to track connections per user and only update presence when the last connection disconnects.

I also learned how to organise Socket.io handlers into separate concerns (chat, rooms, presence) rather than dumping everything into one file — a pattern that keeps the codebase maintainable as features grow.
