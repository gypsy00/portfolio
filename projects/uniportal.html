UNIPortal

A university course advertising website with a public course browser and a protected admin dashboard — built on a custom MVC framework written from scratch in PHP.

Run It Locally

# 1. Clone the repo git clone https://github.com/gypsy00/UNIPortal.git cd UNIPortal # 2. Start the entire stack with Docker docker-compose up --build # 3. Open in your browser # http://localhost:8080 # That's it — Docker handles PHP, Nginx, and MariaDB automatically.

Requires Docker and Docker Compose. One command spins up the PHP-FPM server, Nginx, and MariaDB database — no manual setup needed. Browse courses publicly or log into the admin dashboard to manage content.

The Problem

Universities need a way to showcase their courses to prospective students while giving staff the tools to manage course content, subject areas, and student enquiries — all through a single system.

Rather than using an existing CMS or framework like Laravel, I built the entire MVC architecture from scratch. The goal was to deeply understand how web frameworks work internally — routing, autoloading, templating, ORM patterns, and session-based authentication.

How It Works

All requests hit a single entry point (public/index.php) where Nginx routes everything. The custom router maps URLs to controller actions. A generic DatabaseTable class handles all CRUD operations without writing SQL per feature — you pass it a table name and primary key, and it builds queries dynamically.

Nginx → all requests → public/index.php ↓ Custom Router (URL → Controller::action) ↓ Controllers ├── HomeController (public course browsing) ├── CourseController (CRUD, filtering) ├── SubjectAreaController (CRUD) ├── EnquiryController (view, respond) └── AccountController (login, staff management) ↓ DatabaseTable (generic ORM) ↓ MariaDB via PDO prepared statements ↓ Entity Objects → Templates (.html.php)

Key Features

Public Course Browser Prospective students can browse courses filtered by subject area, type (full-time/part-time), and duration.
Protected Admin Dashboard Staff log in with session-based auth (bcrypt hashed passwords) to manage all content through a clean admin interface.
Subject Area Management Full CRUD for subject areas that categorise courses. Courses are linked to subject areas via foreign keys.
Course Management Add and edit courses with title, description, length, type, modules, and part-time availability.
Enquiry System Students submit enquiries through the public site. Staff can view and respond to them from the dashboard.
Staff Account Management Superadmin can add and delete staff accounts. Passwords are hashed before storage.
Pagination All list views paginate at 10 records using SQL LIMIT/OFFSET — no loading entire tables into memory.

Tech Stack

PHP 8 (Custom MVC) No framework — custom router, autoloader, controllers, entity objects, and template engine built from scratch.
MariaDB / MySQL Relational database with foreign key constraints. All queries use PDO prepared statements to prevent SQL injection.
Docker + Docker Compose Containerised setup with separate PHP-FPM and Nginx containers. One command to spin up the entire stack.
Nginx Web server configured to route all requests through a single PHP entry point for clean URLs.
Custom Autoloader PSR-4 style autoloading with two namespaces: Csy2089 (reusable framework) and Nmp (project-specific code).
HTML / CSS Server-rendered templates with zero JavaScript dependencies. Clean separation — no database logic in views.

What I Learned

Building an MVC framework from scratch gave me a deep understanding of what frameworks like Laravel and Express do under the hood. I had to solve routing, autoloading, database abstraction, session management, and template rendering myself — problems that are usually invisible when using a framework.

The generic DatabaseTable class was the most valuable abstraction. Instead of writing separate SQL for each feature, I built one class that handles INSERT, UPDATE, DELETE, and SELECT for any table. This taught me how ORMs work at a fundamental level.

Containerising with Docker also taught me about environment consistency — the same setup runs identically on any machine, which eliminated "works on my machine" problems during development.
